### STEP 2 · Trigger·Action·조건 분기 미리 설계하기

- **Trigger (시작 이벤트)**: Google Forms – 새 응답 감지 (New Form Response)
- **조건 분기 기준 (Filter/Router)**: 제출된 퀴즈 점수가 80점 이상인 경우 (Path A) / 80점 미만인 경우 (Path B)
- **Action 1**: Google Sheets – 스프레드시트에 행 추가 (Create Spreadsheet Row)
- **Action 2**: Gmail – 합격/불합격 안내 메일 전송 (Send Email)

**Trigger(트리거)·Action(액션) 개념을 내 말로 한 줄씩:**

- **Trigger란**: 워크플로우 전체가 자동으로 작동하도록 방아쇠를 당기는 '특정 이벤트의 발생(시작점)'이다.
- **Action이란**: 트리거가 발동된 이후, 자동화 툴이 연동된 앱을 통해 실제로 수행하는 '구체적인 기능이나 작업'이다.
- **조건 분기(Filter/Router)의 역할**: 유입된 데이터의 특정 조건값(텍스트, 숫자 등)을 검사하여 각기 다른 후속 Action 경로로 흘러가도록 제어하는 댐의 수문 역할을 한다.

### STEP 6 · 비교 분석표 & 최종 의견

| 비교 항목 | 도구 A (Zapier) | 도구 B (Make) |
|---|---|---|
| UI/UX | 리스트 형태의 수직 탑다운 구조로 초보자에게 매우 친숙함 | 비주얼 중심의 캔버스 구조로 자유도가 높고 역동적임 |
| 설정 난이도 | 매우 쉬움 (단계별 필드 가이드가 매우 친절함) | 보통 (함수 사용 및 데이터 타입 매핑 이해 필요) |
| 연동 서비스 범위 | 전 세계 최다 수준 (거의 모든 웹 서비스 연동 가능) | 메이저 서비스는 다 갖췄으나 비주류 서비스는 HTTP 연동 필요 |
| 무료 플랜 범위 | 매우 좁음 (1:1 단순 자동화만 가능, 분기는 유료 필수) | 상당히 관대함 (무료 플랜에서도 라우터 분기 및 다중 단계 지원) |
| 실행 로그 확인 방식 | 텍스트 기반 기록 탭에서 타임라인별 오류 추적 | 캔버스 상에서 각 모듈의 그래픽 버블(Bubble)을 클릭해 추적 |

**도구 A 장단점 (Zapier):**
- 장점: 압도적인 연동성 덕분에 '이 앱도 될까?' 싶은 것까지 다 지원하며, UI가 쉽고 단순해 진입 장벽이 낮습니다.
- 단점: 조금만 정교한 조건문이나 멀티 스텝을 넣으려 해도 유료 결제를 강제하므로 유지 비용 부담이 큽니다.

**도구 B 장단점 (Make):**
- 장점: 무료 플랜에서도 라우터를 통한 유연한 분기 기능과 복잡한 데이터 변환 함수를 제약 없이 쓸 수 있어 비용 효율성이 극대화됩니다.
- 단점: 캔버스형 인터페이스와 매핑 함수 문법 때문에 노코드 툴임에도 약간의 개발적 사고와 초기 학습 곡선이 필요합니다.

**어떤 상황에서 어떤 도구가 적합한지 내 의견:**
단순하고 빠른 1:1 연동이 주를 이루고 예산이 넉넉하거나 개발 지식이 전무한 팀에게는 Zapier가 최고의 선택입니다. 반면 예산을 아끼면서 조건 분기가 복잡하게 얽힌 정교한 비즈니스 워크플로우를 직접 커스텀하여 장기적으로 운영하고 싶다면 Make를 사용하는 것이 훨씬 현명합니다.

**유료 결제가 불가피했던 기능과 무료 대안:**
Zapier에서 조건 분기(Paths) 기능을 설계할 때 결제 팝업이 발생했습니다. 이를 해결하기 위해 무료 플랜에서도 정교한 분기(Router)를 기본 제공하는 Make를 대안으로 선택하여 최종 구현을 완수했습니다.

## 문서 2 · 프로젝트 2: 자유 주제 자동화 설계·구현

### STEP 7 · 자유 주제 반복 업무 하나 정의하기

- **자동화할 반복 업무**: 노션(Notion) 데이터베이스에 새로운 '블로그 콘텐츠 아이디어' 기획안이 등록되면, 생성형 AI가 자동으로 본문 초안 요약과 해시태그를 생성하고, 승인 권한이 있는 담당자 슬랙 채널로 알림 메시지를 발송하는 업무.
- **반복 빈도와 절약 시간**: 주 3회 발행, 회당 평균 20분 소요 → 매주 약 60분(1시간)의 단순 복사·붙여넣기 및 요약 시간이 낭비됨.

### STEP 8 · 도구 1개 선정 및 이유

- **선정 도구**: Make
- **선정 이유**: 노션 API와의 데이터 동기화 안정성이 매우 뛰어나고, 중간에 OpenAI(ChatGPT) 모듈을 삽입하여 텍스트를 정제해야 하는 구조이기 때문입니다. 특히 유료 결제 없이도 '노션 감지 → AI 텍스트 생성 → 슬랙 알림'으로 이어지는 다중 단계(Multi-step) 워크플로우를 유연하게 구축할 수 있는 가장 경제적인 도구라 판단했습니다.

### STEP 9 · 워크플로우 설계

- **Trigger**: Notion – Watch Database Items (새 페이지 생성 및 상태값 변경 감지)
- **조건 분기 기준**: 콘텐츠 카테고리 태그가 '기술/IT'인 경우 / '비즈니스/트렌드'인 경우
- **Action 1**: OpenAI (ChatGPT) – Create a Completion (프롬프트를 통한 초안 요약 및 키워드 추출)
- **Action 2**: Slack – Create a Message (해당 담당 부서 슬랙 채널로 맞춤형 알림 전송)

**워크플로우 흐름:**
1. **Notion 트리거**: 노션 '콘텐츠 기획 시트'에 상태가 '초안 검토'인 신규 페이지가 생성되는지 실시간 모니터링합니다.
2. **Router 분기**: 등록된 글의 카테고리 태그 정보(IT vs 비즈니스)를 읽고 타겟 채널에 맞춰 분기합니다.
3. **OpenAI 액션**: 노션 본문의 핵심 키워드를 ChatGPT API로 전송하여 3줄 요약문과 추천 해시태그 5개를 실시간으로 받아옵니다.
4. **Slack 액션**: 생성된 AI 요약문과 노션 원본 링크를 정돈된 포맷 블록으로 가공하여 담당 부서 슬랙 채널에 즉시 전송합니다.

### STEP 12 · 실행 흐름 최종 정리

1. **Trigger**: 사용자가 노션 데이터베이스에 새로운 블로그 기획 아이디어를 업로드하면 툴이 변경 사항을 감지하여 데이터를 수집합니다.
2. **Action 1**: 수집된 기획안 내용을 바탕으로 OpenAI 모듈이 백엔드에서 작동하여 검색 최적화용 해시태그와 서론 요약본을 자동으로 빌드합니다.

4. **Action 2**: 매핑 결과에 따라 매칭된 최종 슬랙 채널에 AI가 도출한 요약 코멘트와 노션 바로가기 URL을 마크다운 카드로 즉시 전송 및 공유합니다.

### STEP 14 · 보너스 2 — 실패 알림 및 재시도 전략

- **시도 여부**: 시도함

**실패 알림 방식:**
Make 워크플로우 각 모듈 하단에 오류 처리기(Error Handler)인 슬랙 알림 라우터를 추가 설치했습니다. 런타임 오류나 API 할당량 초과 에러가 터지면, 즉시 시스템 관리자 전용 #bot-error-logs 슬랙 채널로 에러 메시지와 에러 코드를 동봉해 긴급 다이렉트 핑을 쏘도록 처리했습니다.

**재시도/대체 경로 설계:**
OpenAI API 서버의 순간적인 순단 현상이나 타임아웃에 대비해 5분 간격으로 총 3회 자동 재시도(Retry) 옵션을 세팅했습니다. 만약 3회 재시도마저 최종 실패할 경우, 전체 프로세스가 뻗지 않도록 Ignore 처리 후 'AI 요약 실패 - 수동 작성 요망'이라는 디폴트 텍스트 문자열을 채워서 슬랙에 안전하게 넘겨주는 대체 경로(Fallback)를 완성했습니다.

### STEP 15 · 만약에? — 세 가지 질문

**무료 플랜 한도를 초과할 것 같다면:**
나는 우선 Make의 데이터 폴링 주기(Polling Interval)를 실시간 분 단위에서 1시간 또는 하루 2회 단위로 늘려 불필요한 자동 오퍼레이션 소모를 최소화하겠다. 또한, 노션 쪽에서 검토 완료된 특정 데이터만 선별하여 트리거하도록 필터를 전면 배치할 것이며, 그럼에도 비즈니스 트래픽 증가로 한도가 넘는다면 이 자동화로 아낀 기회비용과 업무 생산성 향상 지표를 팀원들에게 정량적으로 공유해 공식 유료 플랜으로 전환하겠다.

**연동 서비스가 갑자기 API를 바꾸거나 끊는다면:**
나는 먼저 구축해 둔 실패 알림 채널의 로그를 확인해 어느 단계(노션, OpenAI, 슬랙)의 연결 고리가 끊어졌는지 명확히 파악하겠다. 이후 각 플랫폼의 개발자 공식 독스(Developer Docs) 변경 사항 릴리즈 노트를 추적하여 바뀐 인증 방식이나 JSON 파라미터 구조를 분석할 것이며, Make의 기본 모듈이 업데이트되기 전이라면 직접 HTTP/Webhook 커스텀 모듈을 사용해 변경된 API 명세에 맞춰 긴급 수동 매핑 작업을 진행하겠다.

**이 자동화가 잘못된 데이터를 계속 처리하고 있다면 어떻게 알아챌까:**
나는 데이터가 넘어오는 단계마다 유효성 검증 규칙(Validation Rule) 함수를 심어두어 필수 제목이 비어있거나 이상 문자열이 감지되면 프로세스를 강제 중단(Break)시키겠다. 더불어 슬랙으로 전송되는 모든 자동화 결과물 하단에 '데이터 오류 신고 및 로그 확인' 바로가기 버튼 링크를 고정 배치하여, 실무 담당자들이 업무 도중 가짜 데이터나 오작동을 발견하는 즉시 수동 제어하고 피드백할 수 있는 감시 체계를 만들겠다.
